# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single static marketing page for Ets Lévêque, a heating/plumbing/electrical company in Notre Dame de Sanilhac (Dordogne, France). No framework, no build step, no package manager — plain HTML/CSS/vanilla JS served as-is.

## Running locally

There is no build/lint/test tooling. Serve the directory statically and open it in a browser:

```
npx --yes serve -l 8080 .
```

Repo root regularly accumulates large raw media at top level — source video clips, reference screenshots, AI-generated images, one-off exports (long descriptive names or bare hashes, e.g. `Drone_sequence_shot_property_arc….mp4`, `Screenshot 2026-*.png`) — used to produce real site assets (`assets/video/*.mp4`, `assets/img/photos/*.jpg`, `assets/hero-frames*/frame_NNN.webp`) but not referenced by `index.html`/CSS/JS themselves, and not consistently tracked in git. Don't treat root-level media as live site assets regardless of which specific files are present at any given time; the live assets always live under `assets/`.

### Testing on a real phone (GitHub Codespaces)

`npx serve` only binds locally; reaching it from an actual phone needs either a Cloudflare quick tunnel (`cloudflared tunnel --url http://localhost:8080` — free tier, no login wall, but prone to random edge disconnects/latency under sustained real-device use, not reliable for anything beyond a quick check) or the Codespace's own forwarded port URL (`https://<codespace-name>-8080.app.github.dev`, more stable). **The forwarded port defaults to "Private" visibility**, which puts a GitHub-login wall in front of it — an external device without an authenticated session there gets blocked before the page (or its video/asset requests) ever load, which looks exactly like "nothing works at all" rather than an auth failure, and is easy to misdiagnose as a code or network bug. Set it to public before testing: `gh codespace ports visibility 8080:public --codespace "$CODESPACE_NAME"`.

## Architecture

`index.html` is the only page. It pulls in CSS files individually (one per section/concern, loaded in this order in `<head>`) and eight plain `<script>` tags at the end of `<body>` — no bundler, no modules. Script order matters only in that each one guards its own `document.querySelector` misses (`if (!el) return`), so it's safe for a given script to run even on a page state where its target markup isn't present.

- `css/tokens.css` — design tokens (`:root` custom properties): colors, font stacks, type scale, 8px spacing scale, easing/duration, radius/shadow. Every other CSS file consumes these variables rather than hardcoding values.
- `css/base.css`, `header.css`, `contact.css`, etc. — one stylesheet per section, named to match the section it styles. `header.css` also holds the mobile nav panel's styling (`.mobile-nav*`), not a separate file.
- `css/scrollvid.css` — shared styling for the two scroll-scrubbed video sections (hero + drone).
- `css/stove.css` — styling for the `#poele` section (see video-in-view below).
- `css/gallery-duo.css` — styling for the "Nos services" cards (`#realisations`, see below) — the filename is a holdover from an earlier zigzag-layout version of this section; the classes it defines today are `.service-card*`, not `.gallery-duo__*` (nothing in `index.html` uses the `gallery-duo` name except the outer section's class/id and the stylesheet's filename).
- `js/scroll-video.js` — canvas-based "scroll-scrubbed video" effect (see below). Runs on `DOMContentLoaded` for every `.scrollvid` section.
- `js/stats.js` — builds and animates the big numbers in the "chiffres clés" section (`#chiffres`), see below.
- `js/reveal.js` — generic fade/slide-in-on-scroll for any element with class `.reveal` (same one-shot `IntersectionObserver` pattern).
- `js/video-inview.js` — scroll-locking video playback for any `.video-inview` element (see below). Currently used by the `#poele` (wood stove) section.
- `js/hero-intro.js`, `js/scroll-effects.js`, `js/typewriter.js` — three separate word/character-split text reveal effects used in different sections (see "Text reveal effects" below).
- `js/mobile-nav.js` — opens/closes the full-screen `.mobile-nav` panel from the header burger button.

### Scroll-scrubbed video sections (hero + drone)

The hero and the drone-shot section (`#hero`, `#drone` in `index.html`) aren't real playing videos on desktop — they're an image-sequence flipbook drawn to a `<canvas>`, advanced by scroll position, implemented in `js/scroll-video.js`:

- Each section declares `data-frame-count`, `data-frame-prefix`, `data-frame-ext` and contains a `.scrollvid__sticky` wrapper with a `<canvas>` and a fallback `<video>`.
- **Desktop / no reduced-motion**: the `<video>` element is removed from the DOM entirely; frames (`assets/hero-frames/frame_NNN.webp`, `assets/hero-frames-2/...`) are fetched progressively and decoded to `ImageBitmap` on scroll (rAF-throttled). The hero section (`priority: "high"`) eagerly fetches its first 20 frames on load; every other section (`priority: "low"`, e.g. drone) defers all fetching to `requestIdleCallback` so it never steals bandwidth from the hero during initial page load. Once decoded, bitmaps are kept in memory for the life of the page — **there is no eviction/windowing**; a bounded rolling window was tried and reintroduced stutter without fixing anything, so don't reintroduce one without being explicitly asked (see the comment above `PRIORITY_RADIUS` in `scroll-video.js`). A `PRIORITY_RADIUS` of 12 frames around the current scroll position is force-fetched+decoded symmetrically (both directions) on every scroll tick so fast scrolling — forward or backward — stays smooth even ahead of the background progressive load.
- **Mobile (`max-width: 768px`) or `prefers-reduced-motion: reduce`**: the canvas is removed instead and the native `<video>` takes over — no frame-by-frame decode work. Under `prefers-reduced-motion` it's a plain autoplay/loop. On plain mobile (motion allowed) the two sections behave differently: the hero just plays through once, start to finish, and freezes on its last frame (`loop` deliberately absent from its `<video>` tag in `index.html` — see "end-of-scroll superimposition" below for why); the drone section instead stays paused and is scrubbed by `currentTime = progress * video.duration` on every scroll tick (same progress↔scroll formula as the desktop canvas path), so its text chapters stay in sync with the visible frame in both scroll directions. Before the first `currentTime` write actually takes effect, the drone video is silently "primed" with a `play()` immediately followed by `pause()` — iOS Safari can otherwise ignore `currentTime` seeks on a `<video>` that has never played, leaving the image frozen on frame 0 while the chapter text keeps changing underneath it. A small epsilon (`video.duration - 0.15`) keeps the scrub away from the exact end value, since landing exactly on `duration` can trigger loop/restart behavior in some engines and visibly "superimpose" frame 0 back over the last frame.
- **Mobile autoplay reliability** (`attachNetworkRecovery` / `retryPlayOnGesture` in `scroll-video.js`, mirrored inline in `video-inview.js` for the stove video): two real-device failure modes were confirmed via a temporary on-screen debug overlay (since Safari's console isn't reachable on a phone) and are now handled defensively for every mobile/`.video-inview` video:
  1. **Silently-rejected autoplay** — a muted+`playsinline` `<video>` is normally exempt from autoplay restrictions, but iOS can still block `.play()` outright (Low Power Mode, or the per-site Safari "Auto-Play" setting) with **no error event at all** — the debug overlay showed `readyState` stuck at 1 (metadata only), `paused: true`, forever, with the rejection (`NotAllowedError`) only visible by explicitly inspecting the `play()` promise's rejection reason. A silent `.catch(() => {})` on the initial `play()` call hides this permanently. The fix registers a one-time `touchstart`/`click` listener that retries `play()` on the very next real gesture — which iOS always allows — rather than giving up.
  2. **Mid-stream stalls** (more likely over a flaky tunnel/cellular connection than over solid wifi) — `attachNetworkRecovery` watches for `error`/`stalled` events and a `waiting`-without-progress timeout (5s), and recovers by calling `video.load()` (forces a fresh fetch of the source) then restoring `currentTime` and resuming playback, up to a handful of retries.
  - Separately, on mobile the `<video>` must **not** carry `will-change: transform` / `transform: translateZ(0)` (see the mobile override in `css/scrollvid.css`) — those hints exist only to help the desktop canvas's repeated `drawImage` calls, but on a real iPhone they were observed to force the video onto a GPU layer that iOS then never repaints after an autoplay start: the video keeps decoding normally (`currentTime` advancing, `readyState` reaching 4) while the screen stays visually frozen on its first frame. Confirmed via the same on-device debug overlay; not reproducible in Chromium/Playwright, so don't rely on emulated testing to catch a regression here.
  - The hero video is preloaded via a mobile-only `<link rel="preload" as="video" media="(max-width: 768px)">` in `<head>` (see top of `index.html`) so its ~5MB file starts downloading during initial HTML parsing rather than waiting for `DOMContentLoaded`-driven JS to flip `preload` to `"auto"` — the gap between those two points was large enough on a real device to read as "the video takes too long to start."
- Scroll listeners on both sections skip all decode/draw work when the section is far outside the viewport (see the early-return in `onScroll`), since both sections have independent listeners running simultaneously.
- The drone section also has scroll-linked "chapter" text overlays (`.scrollvid__chapter[data-from]`) driven by the same scroll progress value, with a configurable fade-out window near the end (`data-fade-out-start` / `data-fade-out-end`) so text doesn't get clipped by the next section.
- Changing frame counts/paths means regenerating the whole `assets/hero-frames*/frame_NNN.webp` sequence to match — the JS assumes zero-padded 3-digit filenames starting at `frame_001`.
- Per-chapter `is-active` toggling (`js/scroll-video.js`'s `initChapters`) has **no upper bound** — the last chapter whose `data-from` threshold is crossed stays active until the next one crosses, deliberately, so it doesn't disappear at the very end of the scroll range. This means on reverse-scroll it only deactivates once progress drops back below that chapter's `data-from`, at which point it must disappear **fast**: `.scrollvid__chapter:not(.is-active) .scrollvid__chapter-title .word/-text/-cta` in `drone.css` override the entrance transition with a short, undelayed exit one (150ms, no stagger). Without this, reverse-scrolling past a chapter left its text visibly sitting over the roof for the ~600-850ms entrance-style fade duration, since the video frame (roof vs. wall) moves much faster than that. Don't restore a single shared transition for enter/exit — they're intentionally asymmetric.
- The "Électricité générale" chapter (`[data-from="0.7"]`) is positioned top-right instead of the default bottom-left, with a fixed pixel `width` (currently `500px`) sized to fit its title's `scrollWidth` at the shared `font-size: var(--fs-2xl)` — the title has no internal wrapping safeguard, so if the box is narrower than the widest word's rendered width, the word visually overflows the box (and, at certain viewport aspect ratios, the roof) since nothing clips it. If you change this chapter's copy or font-size, re-measure `title.scrollWidth` and make sure the container `width` comfortably exceeds it. This has been tuned/broken/re-tuned several times chasing roof-overflow bugs across 1173/1440/1920px+ viewports — verify at multiple widths (and note the video's `object-fit: cover` crop shifts what's visible behind the text unpredictably per aspect ratio, so "no overflow at 1920px" doesn't guarantee "no overflow at 1173px").

### Scroll-locked in-view video (`js/video-inview.js`)

Distinct from the scroll-scrubbed sections above: a `.video-inview` element (a real `<video>`, not canvas frames) plays straight through once its section is ~fully in view, and **locks page scroll** until playback ends, then unlocks and reveals a `.video-inview-reveal` overlay. Key details if you touch this:

- Trigger is **not** an `IntersectionObserver` — it's a direct geometry check (`onScroll` comparing `section.getBoundingClientRect()` against the viewport, with a `TOLERANCE` of 60px) against the enclosing `<section>` (found via `video.closest("section")`), since that section is always exactly `100vh`. An earlier `IntersectionObserver({ threshold: 0.98 })` version required hitting a near-single-pixel intersection ratio to fire, which a fast scroll (wheel flick, trackpad momentum) could jump straight over without ever crossing — the section would sail past fully in-view without ever locking. The geometry check with a tolerance band is far harder to skip over in one scroll tick.
- Right before locking, it force-snaps the scroll position to exact alignment (`window.scrollTo({ top, behavior: "instant" })`, computed from the section's own offset) — since the tolerance band means the scroll can be triggered up to `TOLERANCE`px short of perfect alignment, this corrects it so the video is always framed identically regardless of how fast the user was scrolling when it triggered. `behavior: "instant"` is required specifically because `<html>` has `scroll-behavior: smooth` (`base.css`) — `"auto"` would inherit that and animate instead of snapping.
- The lock itself uses the `position: fixed` body technique (save `scrollY`, set `body { position: fixed; top: -${scrollY}px }`, restore via `window.scrollTo` on unlock) rather than plain `overflow: hidden`. `overflow: hidden` alone was found insufficient: trackpad/touch momentum scrolling is sometimes driven by the browser's compositor rather than dispatchable `wheel`/`touchmove` events, so it can keep drifting the scroll position for a few dozen pixels *after* the lock engages, landing the video mis-framed ("blocked too low"). `position: fixed` removes the viewport from any scrollable context at all during the lock, which momentum can't act on. The `wheel`/`touchmove`/`keydown` listeners with `preventDefault()` are kept as a first line of defense but aren't load-bearing on their own anymore.
- Only triggers **once ever** per video (a `triggered` flag removes the scroll listener after the first fire) — after the video finishes and unlocks, the user can freely scroll back up through the section without re-triggering the lock.
- No-ops entirely under `prefers-reduced-motion: reduce`, since the video won't autoplay and locking scroll behind a static poster would trap the user.
- Playback rate is hardcoded to `1.5x` (`PLAYBACK_RATE`) for pacing.
- Has two independent unlock paths — the video's `ended` event, and a `loadedmetadata`-computed timeout fallback (`duration / rate + 2s`) — plus a `play()`-rejection handler that retries once on the next `touchstart`/`click` (same silently-blocked-autoplay issue as the mobile hero/drone videos, see above) before finally giving up and unlocking after a 4s grace window, so a user is never permanently stuck if playback stalls or is blocked by the browser.
- `video.preload` stays `"none"` in the HTML (so it never competes with the hero's fetch on initial page load) but is flipped to `"auto"` in the background via `requestIdleCallback` shortly after load, well before the section is actually scrolled into view — not only at the exact moment of locking. Only starting the fetch at lock-time meant the lock-then-play sequence had a visible stall while the file bought itself enough buffer to actually play, read as "it takes time before it locks and launches."
- The `#poele` section's only content block (`.stove__content.video-inview-reveal`, title "Entretien ramonage" + one description sentence + CTA) reveals via `.is-active` added the instant `video.play()` resolves — i.e. immediately when the video starts, not partway through. An earlier version had a second content block on the right ("Chauffage", separately titled) revealed mid-playback via a generic `data-reveal-at="<progress 0-1>"` mechanism; that block and the `data-reveal-at` plumbing were removed when the section was consolidated to a single left-aligned block, so don't reintroduce `data-reveal-at` without rebuilding that (now-deleted) progress-timeupdate listener in `video-inview.js`.

### "Chiffres clés" numbers (`#chiffres`, `js/stats.js`)

Each `.meter[data-target]` gets its number written into `.meter__readout` as **plain text** (`Number(...).toLocaleString("fr-FR")`), not markup built from individual digit spans:

- An earlier version rendered each digit as its own scrollable column (an odometer/slot-machine effect) so the numbers could roll to their final value. It was scrapped **twice** after real regressions: (1) a `background-clip: text` gradient set on the outer `.meter__readout` doesn't paint through nested `inline-block`/`flex` digit columns, silently rendering the numbers invisible — background-clip:text only works through simple inline descendants (a plain `<span>` like `.meter__unit`), not through boxes that establish their own formatting context; (2) forcing every digit into a fixed `ch`-wide box kills the font's natural letter-spacing/kerning, which reads as "the typography changed" even though the `font-family` didn't. Don't reintroduce a per-digit column structure without being explicitly asked — plain text is the stable, requested state.
- Entrance animation (`.meter.is-visible`) and the continuous scroll parallax are deliberately driven by **different CSS properties** so they never fight: entrance uses `opacity` + `scale` + `filter: blur()` (toggled via the `.is-visible` class from an `IntersectionObserver`), while parallax uses the standalone `translate` property, set continuously from JS (`initParallax` in `stats.js`, rAF-throttled `scroll` listener) with a short `translate` transition for smoothing. Because `translate`/`scale` are independent CSS properties here (not the `transform` shorthand), JS can update one without clobbering the other.
- Hover applies a second, wider gradient (`.meter:hover .meter__readout`) animated via `background-position` (`@keyframes meter-wave`) for a "color sweep" — kept to two stops (ink `--slate` + this meter's own `--accent`, alternating blue/red per `nth-child`) on purpose; an earlier 6-stop red+blue+slate version read as "rainbow" and was rejected.
- `--font-numeral` (`tokens.css`, currently `"Inter"`) is the dedicated display face for these big numbers — deliberately different from `--font-display` (Archivo, used for headings) so the digits read as a distinct "stat" typographic voice. This has been swapped several times (Archivo → Anton → Sora → Inter) chasing a specific reference look; treat the current value as the settled one, not a placeholder.
- `.certs__marquee` (RGE certification badges) sits above `.container` inside `#chiffres`, right under the `#poele` video — it reuses the exact marquee technique from `.partners` (3x duplicated `.certs__set`, CSS `@keyframes` translateX loop, edge mask-image), just under a different class name. The 6 logos (`assets/img/logos/rge-*.png`) were cropped out of a single combined sprite (`RGE-LOGOS-2-e1732697760471.png`) with a one-off Pillow script (not committed) — the sprite is a 3×2 grid; if it ever needs re-cropping there's no build tooling for it, just ad hoc image work. The edge `mask-image` fade zone is `20%/80%` (not the `.partners` marquee's tighter `6%/94%`) — on a narrow mobile viewport the tighter fade left a logo crossing the edge looking abruptly sliced rather than gracefully fading, read as "loading badly"/broken rather than normal infinite-marquee behavior.

### Text reveal effects (`js/hero-intro.js`, `js/scroll-effects.js`, `js/typewriter.js`)

Three independent scripts implement three different word/character-split reveal effects, each scoped to specific classes/elements — they don't share code, but follow the same base pattern: split `textContent` into `<span>`s at DOMContentLoaded (setting `aria-label` on the container first so screen readers still get the whole string), then stagger each span's reveal via a `--i` custom property or a timed `setTimeout` chain, gated by `prefers-reduced-motion` (which skips straight to the fully-revealed state).

- `js/hero-intro.js` — word-splits `.hero__title` and `.hero__subtitle` (CSS-driven stagger via `--i`), plus a separate `initEyebrowTypewriter()` that animates `.hero__eyebrow` with a `steps()` CSS animation sized to the text's actual measured pixel width (not character count, since the font isn't monospace). Both are delayed by a shared `HERO_REVEAL_DELAY_MS` (2200ms) so the hero's opening video frame gets a moment on screen alone before any text arrives. The eyebrow typewriter is skipped entirely on mobile (`max-width: 768px`) since its `white-space: nowrap` would otherwise clip the full "Chauffage · Plomberie · Électricité · Énergies renouvelables" mid-word rather than let it wrap.
- `js/scroll-effects.js` — `initStarsFlip()` char-splits `.reviews__stars` for a CSS 3D-flip-in effect gated by the shared `.reveal`/`IntersectionObserver` class (see `js/reveal.js`), and `initWordBlurReveal()` word-splits any `.word-blur-reveal` element (currently just the "chiffres clés" heading) for a blur-in effect, triggered by its own `IntersectionObserver` at `threshold: 0.4`.
- `js/typewriter.js` — char-splits any `.type-write` element (currently the reviews heading) and reveals characters one at a time via a randomized `setTimeout` chain (`TYPE_MIN_MS`–`TYPE_MAX_MS`) rather than a CSS `steps()` animation, so the cadence reads as more irregular/typing-like; also gated by its own `IntersectionObserver` at `threshold: 0.4`.
- The drone chapters' word-split (`.scrollvid__chapter-title .word-mask > .word`, see scroll-scrubbed video section above) is a fourth, separate case — its spans are hardcoded directly in `index.html` rather than generated by JS, since that reveal is driven by scroll progress (`initChapters` in `scroll-video.js`) rather than an `IntersectionObserver`/timeout.
- If you add a new heading that needs one of these effects, match it to the existing class it's already using elsewhere (`.word-blur-reveal`, `.type-write`) rather than inventing a fifth variant, unless the desired timing/visual genuinely doesn't fit any of the three.

### Mobile nav (`js/mobile-nav.js`, `.mobile-nav*` in `css/header.css`)

The header burger button opens a full-screen `.mobile-nav` overlay panel (own list of links mirroring the desktop nav + dropdown, flattened) rather than an inline dropdown. It has its own dedicated `.mobile-nav__close` button rather than reusing the header burger to toggle closed, because the panel sits visually above the header once open and the header's logo would otherwise show through behind the first link's text. `mobile-nav.js` also closes the panel on any link click and on window resize past 768px (so a resize from mobile to desktop doesn't leave it stuck open behind the now-visible desktop nav).

### "Nos services" cards (`#realisations`, `css/gallery-duo.css`)

Two large image/text cards below "chiffres clés" on a flat `var(--mist)` background, each a self-contained `.service-card` (media + text side by side, `flex` layout, own rounded white card with `box-shadow`) rather than a zigzag grid of rows: card 1 (`#service-sanitaire`) is a real `<video>` (autoplay/muted/loop) with text on the right; card 2 (`#service-pac`, `.service-card--reverse`) puts text on the left and a two-photo crossfade slideshow on the right. This markup/copy/layout ("repris tel quel d'une maquette fournie") replaced an earlier zigzag `.gallery-duo__row` version — see the comment at the top of `gallery-duo.css` — so don't reintroduce the old row/eyebrow-pill/letter-typing structure described in prior docs; it no longer exists in the DOM.

- Each `.service-card` sets its own `--accent`/`--accent-tint` (`nth-of-type(odd)` → blue, `even` → red), driving only the small `.service-card__tag` pill background/text color ("Nos services") — everything else (title, body copy, CTA) is neutral/red regardless of which card it's in.
- `.service-card__media` is intentionally wider than the text column (`flex: 1.6` vs `flex: 1`) at a `3/2` aspect ratio — an earlier narrower/taller ratio made the video and photos look too small for the available card space.
- The CTA on both cards is `.btn.btn-glow` linking to `#contact` and reads "En savoir plus", a deliberate deviation from the sitewide "Devis gratuit" CTA copy (see Button color conventions below) — don't "fix" it back to match other buttons without being asked.
- The slideshow card's two `<img>` crossfade via a CSS `@keyframes` opacity loop (5s cycle, 2.5s stagger, `animation-fill-mode: backwards`) — no JS involved; `prefers-reduced-motion` freezes on the first image.
- On mobile (`max-width: 768px`) the card stacks to a single column (`flex-direction: column`) and title/body font sizes drop to fixed `rem` values — the desktop sizes were tuned for the wide desktop card and were oversized once the card goes full-width on mobile.
- `#service-sanitaire` / `#service-pac` are the anchor ids the header dropdown deep-links to (see Header navigation below) — keep them on these two card elements if the section is restructured again.

### Header navigation (`css/header.css`)

`.site-header` is `position: fixed` and uses `display: grid; grid-template-columns: 1fr auto 1fr;` (logo / nav / actions) specifically so the nav pill (`.site-header__nav`, a red-outlined rounded pill with a dark semi-transparent `background` + `backdrop-filter: blur()`) shrinks to fit only its own links and stays horizontally centered — a `flex: 1` version was tried first and stretched the pill into an oversized oval spanning almost the full header width, which read as an ugly "green band." If you touch header layout, keep the grid approach; logo/actions need `min-width: max-content` and nav links need `white-space: nowrap` to avoid wrapping once the grid's `auto` column gets squeezed on narrower viewports.

- "Nos services" (`.site-header__nav-item`) opens a hover/focus dropdown (`.site-header__dropdown`, shown via `:hover`/`:focus-within`, no JS needed for the open/close itself) listing all 5 trades in a fixed order: Énergies renouvelables, Électricité générale, Entretien Ramonage, Sanitaire — Plomberie, Pompe à chaleur Air-Air. That order was requested explicitly — don't "alphabetize" or reorder it opportunistically.
- The nav is `display: none` under 768px (see the media query above), so this dropdown is desktop-only, consistent with the rest of the nav.
- Two of the five (Énergies renouvelables, Électricité générale) aren't real sections — they're "chapters" that appear as scroll-linked overlays *inside* the `#drone` scroll-scrubbed video (`.scrollvid__chapter[data-from]`, see the scroll-scrubbed video section above). A plain `href="#drone"` would only ever land at the very top of that section, not at the scroll position where that specific chapter is visible. Instead these two links carry `data-scroll-target="#drone"` and `data-scroll-progress="0.12"`/`"0.7"` (matching each chapter's own `data-from`), and `initProgressLinks()` in `js/scroll-video.js` intercepts the click and computes `section.offsetTop + progress * (section.offsetHeight - window.innerHeight)` — the exact same progress↔scroll formula `onScroll` uses to drive the canvas frame — then does a manual `window.scrollTo({ top, behavior: "smooth" })`. If you ever add a 6th dropdown entry for another chapter in a `.scrollvid` section, follow this same pattern rather than a plain anchor.
- The other three link to real elements: `href="#poele"` (existing section id), and `#service-sanitaire` / `#service-pac` — ids added directly on the two `.service-card` elements inside `#realisations` specifically so each trade can be deep-linked to individually, since the section itself only has one shared id.
- **Known interaction, not a bug to "fix" reflexively**: `#poele`'s video (`js/video-inview.js`) locks page scroll for ~8s the first time it scrolls into view (see that section above). Since `<html>` has `scroll-behavior: smooth` and `#poele` sits before `#chiffres`/`#realisations`/`#contact` in the DOM, *any* anchor jump to a section after it — including the pre-existing "Chiffres clés" link, not just the new dropdown entries — passes through `#poele`'s viewport during the scroll animation and triggers its lock on a fresh page load, stranding the visitor there instead of reaching the intended target. Once that video has played once (per page load), every later anchor jump works normally, including into `#service-sanitaire`/`#service-pac`. Fixing this for real would mean changing how `video-inview.js` reacts to programmatic/anchor-driven scrolling (e.g. skipping the lock when the scroll wasn't user-initiated), which hasn't been done — flag it before quietly working around it if asked to fix the dropdown links.

### Contact map (`#contact`, `css/contact.css`)

The Google Maps embed (`.contact__map iframe`, plain `/maps?q=...&output=embed` URL — no API key, no JS Maps SDK) uses the site's normal card language (`border-radius: 24px`, `box-shadow: var(--shadow-soft)`) instead of Google's default look and the site's original thick `2px solid var(--blue)` border. **The map's own colors are intentionally left alone** — a version that recolored the iframe with `filter: grayscale(1) brightness(1.15) contrast(0.9)` plus a `mix-blend-mode: color` blue tint overlay div was built and explicitly rejected ("trop bleu... fais blanc et normal"); don't reintroduce a color filter/tint on this map without being asked again.

### Button color conventions (`css/base.css`, `css/header.css`)

Every button on the site is red (`.btn-red`, `.btn-glow`) **except two specific holdovers that were deliberately reverted back to their original look**: the header's `.btn-pill` ("Devis gratuit") is white (`background: var(--white)`), and the hero's phone number `.btn-ghost` is transparent with a white outline. Both were briefly made red/solid along with everything else, then explicitly reverted because "ça fait trop rouge" in the hero specifically — don't re-red them without being asked again. `.btn-glow` (solid red + red glow `box-shadow`, pill-shaped) is the standard CTA for text sitting over video or plain-white sections that previously had no button — apply that class (not `.btn-red`) for any new CTA in that context, for visual consistency with the rest of the site.

### Design tokens vs. `ets-leveque-plan-refonte.md`

`ets-leveque-plan-refonte.md` is the original French-language design brief/plan for the visual refresh (color system, typography, phase-by-phase execution plan, a ready-to-paste prompt). It proposed a teal/copper palette (`--copper`, `--teal-deep`, etc.). **The palette actually implemented in `css/tokens.css` diverged from that plan**: it's now derived from the real company logo (red `--red: #FF0000` + blue `--blue: #18A1B4`), with `--teal-active` aliased to `--blue`. When touching colors, treat `tokens.css` as the source of truth over the plan doc — the plan describes intent/history, not current state. Typography, spacing scale, and motion tokens from the plan were kept as implemented.

### Content constraints

Real business content (Google reviews, RGE certifications, partner logos, phone/address, activity descriptions) should be preserved verbatim — this is a lead-gen site and the reviews/stats are genuine business data (e.g. "3990 mètres de cuivre déroulés en 2025"), not placeholder copy. Partner logos (Viessmann, Mitsubishi Electric, Legrand, Hargassner, Hoben, BWT) must never be recolored/retouched.

### Conventions

- CSS classes follow BEM-ish naming per component (`.meter__readout`, `.review-card__meta`, `.service-card__media--video`), scoped one file per section.
- Motion respects `prefers-reduced-motion` throughout (see `tokens.css` media query zeroing `--dur-*`, and the explicit checks in `scroll-video.js` / `stats.js`).
- Marquee-style infinite scrollers (partners logos, review columns) are done by duplicating the content block 2-3x in the HTML (`.partners__set[aria-hidden="true"]`, `.reviews__set[aria-hidden="true"]`) and animating with CSS, not JS.
